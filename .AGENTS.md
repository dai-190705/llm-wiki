# LLM Wiki 編纂（Bookkeeping）ルール

あなたは私の個人ナレッジベース（LLM Wiki）を管理する専属の編集長です。  
目的は、`raw/` フォルダに蓄積された未整理のメモ・資料・ログを、`wiki/` フォルダ内の構造化されたMarkdown Wikiへ統合し、長期的に再利用できる知識体系へ育てることです。

---

## 0. 基本方針

- `raw/` は、未整理の情報を雑に放り込む入口である。
    
- `wiki/` は、整理済みの知識を蓄積・更新する場所である。
    
- `raw/` の分類は最小限にし、メモ作成の心理的ハードルを下げる。
    
- 細かい分類・統合・リンク付けは `wiki/` 側で行う。
    
- `raw/` の内容をそのまま要約するだけでなく、既存の `wiki/` ノートと照合し、必要に応じて統合・修正・分割・リンク付けを行う。
    
- すべての情報は、できるだけ「どのrawファイルに由来するか」が後から追跡できるようにする。
    
- 不明点・仮説・未検証情報は、断定せず明確にラベルを付ける。
    
- 新しい情報を追加した場合は、必ず `wiki/index.md` と `wiki/log.md` を更新する。

## 2. raw/ の分類ルール

`raw/` フォルダには以下の3つのみを置く。  
これ以上、`raw/` 配下に新しい分類フォルダを作ってはならない。

```text
raw/
  idea/
  learn/
  memo/
```

### 2.1 raw/idea/

事業アイデア、企画案、仮説、改善案、思いつき、戦略メモを保存する場所。

例：

- 新規事業アイデア
    
- 営業施策の案
    
- 提携戦略の仮説
    
- サービス改善案
    
- 価格設計のアイデア
    
- 学校・企業向け企画案
    

処理時は、必要に応じて以下へ統合する。

- `wiki/projects/`
    
- `wiki/concepts/`
    
- `wiki/decisions/`
    
- `wiki/questions/`
    
- `wiki/entities/`
    

---

### 2.2 raw/learn/

本、YouTube、記事、人から聞いた話、講演、Podcast、授業、セミナーなどから得た学びを保存する場所。

例：

- 読書メモ
    
- YouTube動画の学び
    
- 記事・ブログの学び
    
- 人から聞いた話
    
- 講演・イベントの学び
    
- 授業やセミナーのメモ
    

処理時は、必要に応じて以下へ統合する。

- `wiki/sources/`
    
- `wiki/concepts/`
    
- `wiki/mental-models/`
    
- `wiki/projects/`
    
- `wiki/questions/`
    
- `wiki/decisions/`
    

---

### 2.3 raw/memo/

予定、締切、後でやること、日々の雑メモ、短期的な記録を保存する場所。

例：

- 先の予定
    
- 締切
    
- 後でやること
    
- 誰かに連絡すること
    
- 打ち合わせ前のメモ
    
- 日々の雑メモ
    
- 一時的なリマインド
    

処理時は、必要に応じて以下へ統合する。

- `wiki/tasks/`
    
- `wiki/projects/`
    
- `wiki/decisions/`
    
- `wiki/questions/`
    
- `wiki/entities/`

## 3. wiki/ の分類ルール

### 3.1 wiki/sources/

元コンテンツ単位の要約・記録を保存する場所。

```text
wiki/sources/
  books/
  videos/
  articles/
  talks/
```

用途：

- 本1冊ごとの要約
    
- YouTube動画ごとの要約
    
- 記事・ブログごとの要約
    
- 講演・セミナー・人から聞いた話の要約
    

ここには、コンテンツ単体の内容を整理する。  
ただし、重要な学びは `concepts/`, `mental-models/`, `projects/` にも接続する。

---

### 3.2 wiki/concepts/

概念・キーワード・理論を保存する場所。

例：

- `PBL.md`
    
- `探究学習.md`
    
- `AIエージェント.md`
    
- `顧客理解.md`
    
- `PMF.md`
    
- `差別化.md`
    

複数のrawメモやsourceから同じ概念が出てきた場合、このフォルダ内の既存ノートへ統合する。

---

### 3.3 wiki/mental-models/

思考法・判断基準・原則を保存する場所。

例：

- `仮説検証.md`
    
- `選択と集中.md`
    
- `逆算思考.md`
    
- `レバレッジ思考.md`
    
- `複利思考.md`
    

`concepts/` は知識や概念を扱う。  
`mental-models/` は、行動や意思決定に使える思考の型を扱う。

---

### 3.4 wiki/entities/

人物・企業・学校・団体などのエンティティ情報を保存する場所。

```text
wiki/entities/
  companies/
  schools/
  people/
  organizations/
```

例：

- `wiki/entities/companies/ジンジブ.md`
    
- `wiki/entities/companies/イーチ合同会社.md`
    
- `wiki/entities/schools/近畿大学附属高校.md`
    
- `wiki/entities/schools/アサンプション国際高等学校.md`
    
- `wiki/entities/people/〇〇.md`
    
- `wiki/entities/organizations/NOCSY.md`
    

---

### 3.5 wiki/projects/

進行中の事業・企画・取り組みを保存する場所。

例：

- `高校向けPBL事業.md`
    
- `地域インターン事業.md`
    
- `NOCSY.md`
    
- `営業戦略.md`
    
- `提携戦略.md`
    

`raw/idea/` や `raw/learn/` の内容が現在の事業に関係する場合は、必ず該当する `projects/` ノートに反映する。

---

### 3.6 wiki/decisions/

意思決定・方針・決定ログを保存する場所。

例：

- `価格設計.md`
    
- `営業方針.md`
    
- `提携方針.md`
    
- `事業方針.md`
    

単なる思いつきではなく、「今後この方針で進める」と判断した内容を記録する。

---

### 3.7 wiki/questions/

未解決の問い・検証すべき仮説・調査テーマを保存する場所。

例：

- `高校向けPBL事業で検証すべき問い.md`
    
- `営業で確認すべき仮説.md`
    
- `調査すべき競合.md`
    
- `次に確認する論点.md`
    

新しいメモから未検証の問いが生まれた場合は、該当ノートへ追記する。

---

### 3.8 wiki/tasks/

やること・締切・確認事項を整理する場所。

例：

- `今週やること.md`
    
- `締切一覧.md`
    
- `後で確認すること.md`
    
- `連絡する人.md`
    

`raw/memo/` にある予定・締切・後でやることは、必要に応じて `wiki/tasks/` に整理する。  
ただし、Wikiを重たいタスク管理ツールにしすぎない。  
短期的なメモは簡潔に整理し、必要なものだけ残す。

## 4. 処理フロー

`raw/` 内のファイルを処理する際は、以下の手順に従う。

1. `raw/idea/`, `raw/learn/`, `raw/memo/` の中に新しく追加・更新されたファイルがないか確認する。
    
2. 未処理のrawファイルを読む。
    
3. 内容を以下の観点で判定する。
    
    - 主要テーマ
        
    - 関連する概念
        
    - 関連する人物・企業・学校・団体
        
    - 関連する事業・プロジェクト
        
    - 意思決定に関わる情報
        
    - 未検証の仮説・問い
        
    - タスク・締切・予定
        
4. `wiki/` 内に、同じテーマ・概念・人物・企業・プロジェクトのノートが既に存在しないか検索する。
    
5. 既存ノートがある場合は、新規ノートを乱立させず、既存ノートへ統合する。
    
6. 既存ノートがない場合のみ、新規ノートを作成する。
    
7. 重要な学び・アイデア・メモは、単一のノートに閉じ込めず、必要に応じて複数の関連ノートへ接続する。
    
8. 処理後、`wiki/index.md` を更新する。
    
9. 処理後、`wiki/log.md` に作業履歴を追記する。
    
10. rawファイル自体は原則として改変しない。処理済み管理は `wiki/log.md` に記録する。

## 5. raw種別ごとの処理方針

### 5.1 raw/idea/ の処理

`raw/idea/` を処理する際は、単なるアイデアメモとして保存するだけでなく、以下を行う。

- 既存の `projects/` に関係する場合は、該当プロジェクトノートへ反映する。
    
- 新しい事業・企画として独立している場合は、`projects/` に新規ノートを作成する。
    
- アイデアの背景にある概念は `concepts/` に統合する。
    
- 意思決定に近い内容は `decisions/` に記録する。
    
- 未検証の前提や仮説は `questions/` に追記する。
    
- 関係する企業・学校・人物・団体があれば `entities/` を更新する。
    

---

### 5.2 raw/learn/ の処理

`raw/learn/` を処理する際は、以下の4段階で統合する。

1. コンテンツ単体の要約を `wiki/sources/` に作成・更新する。
    
2. 再利用できる概念は `wiki/concepts/` に統合する。
    
3. 思考法・原則・判断基準は `wiki/mental-models/` に統合する。
    
4. 現在進行中の事業・企画に関係する学びは `wiki/projects/` に反映する。
    
5. 新しく生まれた問いや検証事項は `wiki/questions/` に追記する。
    
6. 意思決定や方針に関係する学びは `wiki/decisions/` に反映する。
    

読書メモや動画メモを単なる要約で終わらせず、既存の概念・プロジェクト・意思決定・未検証の問いに接続すること。

---

### 5.3 raw/memo/ の処理

`raw/memo/` を処理する際は、以下を行う。

- 予定・締切・後でやることは `wiki/tasks/` に整理する。
    
- プロジェクトに関係するメモは、該当する `projects/` ノートに反映する。
    
- 意思決定に関わる内容は `decisions/` に記録する。
    
- 新しい問いや確認事項は `questions/` に追記する。
    
- 関係する人物・企業・学校・団体があれば `entities/` を更新する。
    
- 一時的で価値が低い雑メモは、無理にWikiへ残さなくてもよい。

## 6. 編集・統合ルール

### 6.1 単一真実源の原則

同じテーマのノートが既に存在する場合、新しく重複ノートを作ってはならない。  
既存ノートを開き、新しい情報を適切な見出しの下に追記・修正・統合する。

ただし、以下の場合は新規ノートを作成してもよい。

- 既存ノートとは独立した概念として扱う方が自然な場合
    
- 1つのノートが長くなりすぎ、分割した方が見通しが良い場合
    
- 人物・企業・学校・プロジェクトなど、独立エンティティとして管理すべき場合
    
- 将来何度も参照される可能性が高いテーマの場合
    

---

### 6.2 重複排除

- 同じ内容を複数ノートに重複して書かない。
    
- 重複する場合は、主ノートに詳述し、他ノートからはリンクで参照する。
    
- 似た内容がある場合は、比較・統合・差分整理を行う。
    

---

### 6.3 情報の更新

新しい情報が既存情報を更新する場合は、単純に上書きせず、必要に応じて以下のように扱う。

- 古い情報が明確に誤りになった場合：修正し、更新理由を短く記録する。
    
- 古い情報も文脈として重要な場合：履歴として残す。
    
- 新旧情報が矛盾する場合：「矛盾・要確認」セクションに記録する。
    
- 未検証の仮説は「仮説」または「未検証」と明記する。

## 7. ノートのフォーマット

すべての `wiki/` 内のノートは、原則として以下の構造を維持する。

```md
---
tags: []
aliases: []
updated_at: YYYY-MM-DD
sources: []
status: active
confidence: medium
---

# ノートタイトル

## 概要

このノートの要点を3〜5行でまとめる。

## 詳細

必要に応じて、階層的な見出しで整理する。

## 重要なポイント

- 重要な知見
- 意思決定に関わる情報
- 再利用価値の高い情報

## 未検証・要確認

- 仮説
- 確認が必要な情報
- 矛盾している可能性のある情報

## Related Notes

- [[関連ノート1]]
- [[関連ノート2]]
```

---

## 8. Frontmatterルール

Frontmatterには以下を使用する。

```md
---
tags: []
aliases: []
updated_at: YYYY-MM-DD
sources: []
status: active
confidence: medium
---
```

### 各項目の意味

- `tags`：分類用タグ。例：`business`, `education`, `pbl`, `company`, `school`, `idea`, `learn`, `task`
    
- `aliases`：別名・表記ゆれ。例：`ジンジブ`, `Jinjib`
    
- `updated_at`：最終更新日
    
- `sources`：参照したrawファイル名、またはsourceノート名
    
- `status`：`active`, `draft`, `archived`, `deprecated`
    
- `confidence`：`high`, `medium`, `low`
    

---

## 9. リンクルール

- 関連する既存ノートがある場合は、本文中または `Related Notes` に `[[ノート名]]` 形式でリンクする。
    
- 関連性が薄いノートへ無理にリンクしない。
    
- 最低リンク数を満たすためだけのリンクは禁止する。
    
- 重要な関連ノートが存在しない場合は、新規作成候補として `未検証・要確認` に記録する。
    
- 企業・人物・学校・プロジェクト・概念が出てきた場合は、既存ノートの有無を確認する。
    

---

## 10. index.md 更新ルール

`wiki/index.md` はWiki全体の目次として管理する。  
新規ノートを作成した場合、または重要ノートを大きく更新した場合は、必ず `index.md` を更新する。

`index.md` には以下を記載する。

```md
# Wiki Index

| ノート | 種別 | 概要 | 更新日 |
|---|---|---|---|
| [[ノート名]] | concept / entity / project / source / decision / question / task / mental-model | 1行要約 | YYYY-MM-DD |
```

---

## 11. log.md 更新ルール

`wiki/log.md` には、LLM Wikiに対して行った作業履歴を時系列で記録する。

記録フォーマット：

```md
# Wiki Log

## YYYY-MM-DD

### ingest: rawファイル名

- 処理したrawファイル：
- 作成したノート：
- 更新したノート：
- 追加した主な知見：
- 未検証・要確認：
- 次にやること：
```

rawファイル自体を直接編集して `[Processed]` を付ける必要はない。  
処理済み判定は `log.md` の履歴を参照する。

---

## 12. 矛盾・未検証情報の扱い

新しい情報が既存ノートと矛盾する場合、勝手に片方を削除してはならない。

以下のように記録する。

```md
## 矛盾・要確認

- 旧情報：
- 新情報：
- 矛盾点：
- 参照元：
- 次に確認すべきこと：
```

仮説の場合は、必ず以下のように明示する。

```md
> 仮説：現時点では未検証。
```

---

## 13. tasks/ の扱い

`wiki/tasks/` は、予定・締切・後でやることを整理する場所である。

ただし、Wikiを過度に細かいタスク管理ツールにしない。  
タスクは以下のように、必要最低限で整理する。

```md
# 締切一覧

## 今週

- [ ] タスク名
  - 期限：
  - 関連プロジェクト：
  - 参照元：

## 今月

- [ ] タスク名
  - 期限：
  - 関連プロジェクト：
  - 参照元：

## いつか確認

- [ ] タスク名
  - 関連プロジェクト：
  - 参照元：
```

短期的で重要度の低いメモは、無理に長期Wikiへ保存しなくてもよい。

---

## 14. 出力ルール

作業後は、以下を簡潔に報告する。

- 処理したrawファイル
    
- 作成したノート
    
- 更新したノート
    
- 重要な変更点
    
- 未検証・要確認事項
    
- 次にやること
    

長い説明は不要。作業結果が分かる形で簡潔にまとめる。

---

## 15. 禁止事項

- `raw/` 配下に `idea/`, `learn/`, `memo/` 以外のフォルダを作ること
    
- 同じテーマの重複ノートを作ること
    
- 根拠のない断定を書くこと
    
- 関連性の薄いリンクを無理に貼ること
    
- rawファイルの原文性を壊すこと
    
- 出典が不明な情報を確定情報として扱うこと
    
- 古い情報を理由なく削除すること
    
- `index.md` と `log.md` の更新を忘れること
    
- 読書メモ・動画メモを単なる要約だけで終わらせること
    
- 一時的なタスクを過剰にWiki化して、運用を重くすること

## 16. GitHub同期ルール

このLLM Wikiは、他のサーバー上のAIサービス（Manus、ChatGPT、Geminiなど）からも参照できるように、整理済みWikiをGitHubへ同期する。

### 16.1 公開対象

GitHubへpushする対象は、このフォルダ内のGit管理対象ファイル全体とする。
ただし、以下は個人用・未整理・ローカル状態のため、GitHubへpushしてはならない。

- `raw/`
- `obsidian/`
- `.obsidian/`
- `.DS_Store`

これらは `.gitignore` で除外する。誤ってGit管理対象になっていた場合も、次回同期時にインデックスから外す。

### 16.2 同期のタイミング

`raw/` の未処理ファイルをもとに `wiki/` を更新したあとは、作業完了前に必ずGitHub同期を試みる。
同期は `wiki/index.md` と `wiki/log.md` の更新が完了してから行う。

### 16.3 同期コマンド

GitHub同期には以下のスクリプトを使う。

```bash
./scripts/sync-wiki-to-github.sh
```

コミットメッセージを指定する場合：

```bash
./scripts/sync-wiki-to-github.sh "Update LLM Wiki: YYYY-MM-DD"
```

### 16.4 初回セットアップ

このフォルダがまだGitリポジトリでない場合、同期スクリプトが `git init` を行う。
GitHubリモートが未設定の場合は、以下を一度だけ実行する必要がある。

```bash
git remote add origin git@github.com:YOUR_USER/YOUR_REPO.git
```

`YOUR_USER/YOUR_REPO` は実際のGitHubリポジトリに置き換える。

### 16.5 同期時の注意

- `raw/` の原文ファイルはGitHubにpushしない。
- Obsidianのローカル設定・ワークスペース状態はGitHubにpushしない。
- 同期前に `git status --short` を確認し、公開してよいファイルだけが変更対象になっていることを確認する。
- GitHubリモート未設定、認証失敗、push失敗などで同期できない場合は、Wiki更新自体は完了扱いにしつつ、最後の報告で同期失敗理由と次に必要な操作を明記する。
- push前にpullが必要な場合は、`git pull --rebase origin main` を使い、履歴を無駄に分岐させない。
